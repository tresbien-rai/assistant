P0-02: Database schema and data access layer
--------------------------------------------------------------------------------
Phase:        0.1
Branch:       agent/P0-02-database-schema
Dependencies: P0-01

Description:
  Set up the SQLite database with better-sqlite3 and create all tables
  needed for Phase 0. Build a data access layer that abstracts all database
  operations behind clean functions, so swapping SQLite for PostgreSQL later
  only requires changing this one module.

  Create:
    server/src/db/
      connection.js     (initialize SQLite, run migrations)
      schema.sql        (all CREATE TABLE statements)
      dal.js            (data access layer - exported functions)

  Database file location: server/data/assistant.db
  (The data/ directory should be gitignored)

  Tables (all use TEXT UUIDs for primary keys via crypto.randomUUID):

    users
      id              TEXT PRIMARY KEY
      google_id       TEXT UNIQUE NOT NULL
      email           TEXT
      display_name    TEXT
      drive_token      TEXT          (encrypted)
      drive_refresh    TEXT          (encrypted)
      created_at      INTEGER NOT NULL  (Unix ms timestamp)
      updated_at      INTEGER NOT NULL

    personas
      id              TEXT PRIMARY KEY
      user_id         TEXT NOT NULL REFERENCES users(id)
      name            TEXT NOT NULL
      system_prompt   TEXT DEFAULT ''
      prefill         TEXT DEFAULT ''
      avatar_filename TEXT DEFAULT ''
      expressions     TEXT DEFAULT '{}'  (JSON string)
      model_config    TEXT DEFAULT '{}'  (JSON string)
      created_at      INTEGER NOT NULL
      updated_at      INTEGER NOT NULL

    conversations
      id              TEXT PRIMARY KEY
      user_id         TEXT NOT NULL REFERENCES users(id)
      persona_id      TEXT REFERENCES personas(id)
      project_id      TEXT            (nullable, for Phase 1)
      title           TEXT DEFAULT 'New Chat'
      created_at      INTEGER NOT NULL
      updated_at      INTEGER NOT NULL

    messages
      id              TEXT PRIMARY KEY
      conversation_id TEXT NOT NULL REFERENCES conversations(id)
      role            TEXT NOT NULL    ('user' or 'assistant')
      content         TEXT DEFAULT ''
      attachments     TEXT DEFAULT '[]' (JSON string)
      created_at      INTEGER NOT NULL

    projects (created now for schema completeness, used in Phase 1)
      id              TEXT PRIMARY KEY
      user_id         TEXT NOT NULL REFERENCES users(id)
      name            TEXT NOT NULL
      instructions    TEXT DEFAULT ''
      drive_folder_id TEXT DEFAULT ''
      created_at      INTEGER NOT NULL
      updated_at      INTEGER NOT NULL

    project_files (created now, used in Phase 1)
      id              TEXT PRIMARY KEY
      project_id      TEXT NOT NULL REFERENCES projects(id)
      filename        TEXT NOT NULL
      mime_type       TEXT DEFAULT ''
      size_bytes      INTEGER DEFAULT 0
      drive_file_id   TEXT DEFAULT ''
      created_at      INTEGER NOT NULL

    settings
      id              TEXT PRIMARY KEY
      user_id         TEXT UNIQUE NOT NULL REFERENCES users(id)
      avatar_size     TEXT DEFAULT 'medium'
      avatar_position TEXT DEFAULT 'top-right'
      show_avatar     INTEGER DEFAULT 1
      custom_models   TEXT DEFAULT '{}' (JSON string)
      created_at      INTEGER NOT NULL
      updated_at      INTEGER NOT NULL

    api_keys
      id              TEXT PRIMARY KEY
      user_id         TEXT NOT NULL REFERENCES users(id)
      provider        TEXT NOT NULL   ('anthropic', 'google', 'openai')
      encrypted_key   TEXT NOT NULL
      created_at      INTEGER NOT NULL
      updated_at      INTEGER NOT NULL
      UNIQUE(user_id, provider)

  Data access layer (dal.js) should export functions like:
    users:
      findUserByGoogleId(googleId)
      createUser({ googleId, email, displayName, driveToken, driveRefresh })
      updateUserDriveTokens(userId, { driveToken, driveRefresh })

    personas:
      getPersonasByUser(userId)
      getPersonaById(personaId, userId)
      createPersona(userId, data)
      updatePersona(personaId, userId, data)
      deletePersona(personaId, userId)

    conversations:
      getConversationsByUser(userId, { personaId?, limit?, offset? })
      getConversationById(conversationId, userId)
      createConversation(userId, data)
      updateConversation(conversationId, userId, data)
      deleteConversation(conversationId, userId)

    messages:
      getMessagesByConversation(conversationId, userId)
      createMessage(conversationId, data)
      updateMessage(messageId, data)
      deleteMessage(messageId)

    settings:
      getSettingsByUser(userId)
      upsertSettings(userId, data)

    apiKeys:
      getApiKey(userId, provider)
      upsertApiKey(userId, provider, encryptedKey)
      deleteApiKey(userId, provider)
      getApiKeyProviders(userId)  (list which providers have keys)

  Every function that takes userId should use it in the WHERE clause
  to enforce data isolation between users.

Acceptance criteria:
  - Server starts and creates the database file if it doesn't exist
  - All tables are created on first run
  - DAL functions can be imported and called (write a small test script
    that creates a user, creates a persona, creates a conversation with
    messages, and reads them back)
  - All DAL functions enforce user_id scoping

Files created:
  - server/src/db/connection.js
  - server/src/db/schema.sql
  - server/src/db/dal.js
  - server/data/.gitkeep

P0-03: Error handling middleware and structured logging
--------------------------------------------------------------------------------
Phase:        0.1
Branch:       agent/P0-03-error-handling
Dependencies: P0-02

Description:
  Add centralized error handling and structured logging to the Express server.
  This must be in place before building API routes so all routes benefit
  from consistent error responses.

  Create:
    server/src/middleware/errorHandler.js
    server/src/utils/logger.js
    server/src/utils/AppError.js

  AppError class:
    - Extends Error
    - Properties: statusCode, code (string like 'PROVIDER_ERROR'),
      message (user-facing), details (optional additional info)
    - Static factory methods for common errors:
        AppError.auth(message)         -> 401, AUTH_ERROR
        AppError.provider(message, details) -> 502, PROVIDER_ERROR
        AppError.drive(message)        -> 502, DRIVE_ERROR
        AppError.rateLimited(retryAfter) -> 429, RATE_LIMITED
        AppError.validation(message)   -> 400, VALIDATION_ERROR
        AppError.notFound(resource)    -> 404, NOT_FOUND
        AppError.server(message)       -> 500, SERVER_ERROR

  Error handler middleware:
    - Catches all errors thrown in routes
    - If error is AppError, return structured JSON response:
        { error: { code, message, details } }
    - If error is unexpected, log full stack trace, return generic
      SERVER_ERROR to client (don't leak internals)
    - Log all errors with appropriate level (warn for 4xx, error for 5xx)

  Logger (using pino for performance):
    - Structured JSON logging in production
    - Pretty-printed in development
    - Log levels: error, warn, info, debug
    - Include request ID, user ID (when available), timestamp
    - NEVER log API keys, tokens, or passwords (create a sanitize utility)

  Also create a request logging middleware:
    - Log method, path, status code, response time for every request
    - Use pino-http or similar

  Wire into index.js:
    - Request logging middleware (first)
    - Error handler middleware (last, after all routes)

  Dependencies to install:
    - pino
    - pino-pretty (dev dependency)
    - pino-http

Acceptance criteria:
  - All requests are logged with method, path, status, response time
  - Throwing AppError.validation('test') in a route returns:
    { "error": { "code": "VALIDATION_ERROR", "message": "test" } }
    with status 400
  - Throwing a raw Error returns a generic SERVER_ERROR with status 500
    and the full stack trace appears in server logs but NOT in the response
  - No sensitive data appears in any log output

Files created:
  - server/src/middleware/errorHandler.js
  - server/src/utils/logger.js
  - server/src/utils/AppError.js

Files modified:
  - server/src/index.js (wire in middleware)
  - server/package.json (new dependencies)

P0-04: Authentication (Google OAuth + JWT sessions)
--------------------------------------------------------------------------------
Phase:        0.4
Branch:       agent/P0-04-auth
Dependencies: P0-03

Description:
  Implement Google OAuth 2.0 login flow that simultaneously authenticates
  the user and obtains Google Drive access tokens. Issue JWT tokens for
  session management.

  Create:
    server/src/routes/auth.js
    server/src/middleware/authenticate.js
    server/src/utils/encryption.js

  Encryption utility:
    - encrypt(text) / decrypt(ciphertext) using AES-256-GCM
    - Uses ENCRYPTION_KEY from environment
    - Used for Drive tokens and API keys stored in the database

  OAuth flow:
    GET /api/auth/google
      - Generates Google OAuth URL with scopes:
        openid, email, profile, https://www.googleapis.com/auth/drive.file
      - Includes state parameter (CSRF protection)
      - Redirects user to Google consent screen

    GET /api/auth/google/callback
      - Receives authorization code from Google
      - Exchanges code for access token + refresh token
      - Fetches user profile (Google ID, email, display name)
      - Creates or updates user in database:
        - If google_id exists: update Drive tokens
        - If new: create user, create default persona, create default
          settings
      - Encrypts Drive tokens before storing
      - Issues JWT containing: { userId, email, displayName }
      - Redirects to frontend with JWT (as query param or set as
        httpOnly cookie, depending on deployment constraints)

    GET /api/auth/me
      - Returns current user info from JWT
      - Used by frontend to check if session is valid

    POST /api/auth/logout
      - Invalidate session (if using cookie, clear it)

  Authenticate middleware:
    - Extracts JWT from Authorization header (Bearer token) or cookie
    - Verifies JWT signature and expiration
    - Attaches user object to req.user (with userId, email, displayName)
    - Returns 401 AUTH_ERROR if invalid/expired

  JWT configuration:
    - Secret from JWT_SECRET env var
    - Expiration: 7 days
    - Include userId, email, displayName in payload

  Google OAuth setup notes (for manual reference, not automated):
    - Requires Google Cloud Console project with OAuth consent screen
    - Credentials: OAuth 2.0 Client ID (Web application type)
    - Authorized redirect URI: {server_url}/api/auth/google/callback

  Dependencies to install:
    - jsonwebtoken
    - googleapis (Google APIs Node.js client)

Acceptance criteria:
  - GET /api/auth/google redirects to Google's consent screen
  - After Google consent, callback creates user and redirects with JWT
  - GET /api/auth/me with valid JWT returns user info
  - GET /api/auth/me without JWT returns 401
  - Drive tokens are encrypted in the database (not plaintext)
  - Second login for same Google account updates tokens, doesn't create
    duplicate user
  - Default persona and settings are created for new users

Files created:
  - server/src/routes/auth.js
  - server/src/middleware/authenticate.js
  - server/src/utils/encryption.js

Files modified:
  - server/src/index.js (mount auth routes)
  - server/package.json (new dependencies)

P0-05: API keys CRUD (encrypted storage)
--------------------------------------------------------------------------------
Phase:        0.3
Branch:       agent/P0-05-api-keys
Dependencies: P0-04

Description:
  Build API endpoints for managing per-provider API keys. Keys are encrypted
  at rest and never returned to the frontend in plaintext.

  Create:
    server/src/routes/apiKeys.js

  Endpoints:
    GET /api/api-keys
      - Returns list of providers that have keys stored
      - Response: [{ provider: "anthropic", hasKey: true, updatedAt: ... }]
      - NEVER returns the actual key values

    PUT /api/api-keys/:provider
      - Body: { key: "sk-ant-..." }
      - Validates provider is one of: anthropic, google, openai
      - Encrypts key using encryption utility from P0-04
      - Upserts into api_keys table
      - Returns: { provider, hasKey: true, updatedAt }

    DELETE /api/api-keys/:provider
      - Removes the stored key for that provider
      - Returns: { provider, hasKey: false }

  Internal helper (not an API endpoint, used by chat proxy later):
    getDecryptedApiKey(userId, provider)
      - Reads encrypted key from DB, decrypts, returns plaintext
      - Throws AppError.notFound if no key stored for that provider

  All routes require authentication (authenticate middleware).

Acceptance criteria:
  - PUT stores a key, GET confirms it exists without revealing it
  - DELETE removes the key, GET confirms it's gone
  - Stored keys in DB are encrypted (inspect DB to verify)
  - Invalid provider names return VALIDATION_ERROR
  - Routes return 401 without valid JWT

Files created:
  - server/src/routes/apiKeys.js

Files modified:
  - server/src/index.js (mount routes)

P0-09: Chat proxy - Anthropic
--------------------------------------------------------------------------------
Phase:        0.2
Branch:       agent/P0-09-chat-anthropic
Dependencies: P0-05

Description:
  Build the chat proxy that forwards requests to Anthropic's API using
  the user's stored API key. Supports both non-streaming and streaming.

  Create:
    server/src/routes/chat.js
    server/src/providers/anthropic.js

  Chat route:
    POST /api/chat
      - Body: { provider, model, messages, systemPrompt, modelParams,
               prefill?, attachments? }
      - Retrieves the user's decrypted API key for the specified provider
      - Delegates to the appropriate provider module
      - Returns the provider's response

    POST /api/chat/stream
      - Same body as /api/chat
      - Sets response headers for SSE (text/event-stream)
      - Pipes the provider's streaming response to the client
      - Handles client disconnect (abort upstream request)

    GET /api/models/:provider
      - Fetches available models from the provider's models endpoint
      - Uses the user's API key
      - Returns the model list

  Anthropic provider module:
    - Builds the request body matching the current app.js format:
      model, max_tokens, system, messages, temperature, top_p, top_k,
      stop_sequences, thinking (if enabled)
    - Handles attachments (multimodal content blocks)
    - For non-streaming: POST to https://api.anthropic.com/v1/messages
      Returns the response text
    - For streaming: POST with stream:true, pipe SSE events back
    - Proper error handling: map Anthropic error responses to AppError
      (e.g., 429 -> AppError.provider with rate limit details)
    - No 'anthropic-dangerous-direct-browser-access' header needed
      (server-side requests don't need CORS workaround)

  The chat.js route should use a provider dispatch pattern:
    const providers = { anthropic, google, openai };
    providers[provider].chat(params) / providers[provider].stream(params)
  This makes adding new providers simple.

Acceptance criteria:
  - Non-streaming chat works: send a message, get a response
  - Streaming chat works: SSE events flow to the client
  - API key is never sent to or visible from the frontend
  - Anthropic error responses are mapped to structured AppError responses
  - Model list endpoint returns available models
  - Client disconnecting mid-stream aborts the upstream request
  - Prefill is correctly handled

Files created:
  - server/src/routes/chat.js
  - server/src/providers/anthropic.js

Files modified:
  - server/src/index.js (mount routes)
